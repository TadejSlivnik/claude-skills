# Frontend UI Engineering

## Overview

Build production-quality user interfaces that are accessible, performant, and visually polished. The goal is UI that looks like it was built by a design-aware engineer at a top company — not like it was generated by an AI. This means real design system adherence, proper accessibility, thoughtful interaction patterns, and no generic "AI aesthetic."

The principles here are framework-agnostic. They apply whether you build with React, Vue, Svelte, Angular, SolidJS, Web Components, or plain HTML/CSS/JS, and whether you style with Tailwind, CSS Modules, vanilla CSS, or a CSS-in-JS library.

> **About the code examples:** Examples are written in React/JSX with utility-class CSS because that's a compact, widely-readable way to show the idea. They are *illustrations of the principle, not a mandate to use that stack.* Each example notes the underlying concept so you can translate it to your project's framework. Always match the conventions already present in the codebase you're working in.

## When to Use

- Building new UI components or pages
- Modifying existing user-facing interfaces
- Implementing responsive layouts
- Adding interactivity or state management
- Fixing visual or UX issues

## Component Architecture

### File Structure

Colocate everything related to a component — the implementation, its tests, its styles, its local state logic, and its component-specific types. The exact filenames and extensions depend on your framework, but the principle is the same: a reader should find everything about a component in one place.

```
# React example — translate the suffixes to your stack
#   Vue:     TaskList.vue, TaskList.spec.ts, useTaskList.ts
#   Svelte:  TaskList.svelte, TaskList.test.ts
#   Angular: task-list.component.ts/html/scss/spec.ts
src/components/
  TaskList/
    TaskList.tsx          # Component implementation
    TaskList.test.tsx     # Tests
    TaskList.stories.tsx  # Component stories / docs (if using)
    use-task-list.ts      # Extracted state logic (if complex)
    types.ts              # Component-specific types (if needed)
```

### Component Patterns

**Prefer composition over configuration.** Expose structure through nested, composable pieces rather than an ever-growing list of configuration props. (In React this is `children`; in Vue/Svelte/Angular it's slots / content projection; in Web Components it's `<slot>`.)

```tsx
// Good: Composable
<Card>
  <CardHeader>
    <CardTitle>Tasks</CardTitle>
  </CardHeader>
  <CardBody>
    <TaskList tasks={tasks} />
  </CardBody>
</Card>

// Avoid: Over-configured
<Card
  title="Tasks"
  headerVariant="large"
  bodyPadding="md"
  content={<TaskList tasks={tasks} />}
/>
```

**Keep components focused** — a component should do one thing. If it renders a list *and* fetches data *and* manages a modal, split it.

```tsx
// Good: Does one thing (renders a single task row)
export function TaskItem({ task, onToggle, onDelete }: TaskItemProps) {
  return (
    <li className="flex items-center gap-3 p-3">
      <Checkbox checked={task.done} onChange={() => onToggle(task.id)} />
      <span className={task.done ? 'line-through text-muted' : ''}>{task.title}</span>
      <Button variant="ghost" size="sm" onClick={() => onDelete(task.id)}>
        <TrashIcon />
      </Button>
    </li>
  );
}
```

**Separate data fetching from presentation.** Keep a "container" that owns data, loading, and error concerns, and a "presentation" piece that only renders what it's given. This separation is universal — the names differ (container/presentational, smart/dumb, route component/view) but the boundary is the same, and it makes the presentation layer trivial to test and reuse.

```tsx
// Container: handles data, loading, and error/empty branches
export function TaskListContainer() {
  const { tasks, isLoading, error, refetch } = useTasks();

  if (isLoading) return <TaskListSkeleton />;
  if (error) return <ErrorState message="Failed to load tasks" retry={refetch} />;
  if (tasks.length === 0) return <EmptyState message="No tasks yet" />;

  return <TaskList tasks={tasks} />;
}

// Presentation: handles rendering only, given fully-resolved data
export function TaskList({ tasks }: { tasks: Task[] }) {
  return (
    <ul role="list" className="divide-y">
      {tasks.map(task => <TaskItem key={task.id} task={task} />)}
    </ul>
  );
}
```

## State Management

**Choose the simplest approach that works.** Match state to its scope and lifecycle rather than reaching for a global store by default. The categories below are universal; the named tools are just common examples per ecosystem.

| State category | When to use it | Common tools |
|---|---|---|
| Local component state | UI state owned by one component | React `useState`, Vue `ref`, Svelte `$state`, Angular signals |
| Lifted / shared state | Shared between 2–3 sibling components | Lift to the nearest common parent and pass down |
| App-wide read-heavy state | Theme, auth, locale (read often, written rarely) | React Context, Vue `provide`/`inject`, Svelte context, Angular DI |
| URL state | Filters, pagination, anything shareable/bookmarkable | Router query/search params |
| Server state | Remote data with caching, refetch, invalidation | TanStack Query, SWR, RTK Query, Apollo |
| Global client store | Complex client state shared across the whole app | Zustand, Redux, Pinia, Svelte stores, NgRx |

A key distinction that holds in every framework: **server state is not client state.** Cache, revalidate, and dedupe remote data with a dedicated data-fetching layer instead of hand-rolling it inside a global store.

**Avoid prop drilling deeper than ~3 levels.** If you're threading props through components that don't use them, introduce a context/provide mechanism or restructure the tree.

## Design System Adherence

### Avoid the AI Aesthetic

AI-generated UI has recognizable patterns. Avoid all of them — these apply regardless of framework or styling approach:

| AI Default | Why It Is a Problem | Production Quality |
|---|---|---|
| Purple/indigo everything | Models default to visually "safe" palettes, making every app look identical | Use the project's actual color palette |
| Excessive gradients | Gradients add visual noise and clash with most design systems | Flat or subtle gradients matching the design system |
| Rounded everything (maximum border radius) | Maximum rounding signals "friendly" but ignores the hierarchy of corner radii in real designs | Consistent border-radius from the design system |
| Generic hero sections | Template-driven layout with no connection to the actual content or user need | Content-first layouts |
| Lorem ipsum-style copy | Placeholder text hides layout problems that real content reveals (length, wrapping, overflow) | Realistic placeholder content |
| Oversized padding everywhere | Equal generous padding destroys visual hierarchy and wastes screen space | Consistent spacing scale |
| Stock card grids | Uniform grids are a layout shortcut that ignores information priority and scanning patterns | Purpose-driven layouts |
| Shadow-heavy design | Layered shadows add depth that competes with content and slows rendering on low-end devices | Subtle or no shadows unless the design system specifies |

### Spacing and Layout

Use a consistent spacing scale. Don't invent values — whether you express spacing as utility classes, design tokens, CSS custom properties, or raw CSS, every value should come from the scale.

```css
/* Use the scale: e.g. 0.25rem increments (or whatever the project defines) */
/* Good */  padding: 1rem;      /* 16px — on scale */
/* Good */  gap: 0.75rem;       /* 12px — on scale */
/* Bad */   padding: 13px;      /* Not on any scale */
/* Bad */   margin-top: 2.3rem; /* Not on any scale */
```

### Typography

Respect the type hierarchy:

```
h1 → Page title (one per page)
h2 → Section title
h3 → Subsection title
body → Default text
small → Secondary/helper text
```

Don't skip heading levels. Don't use heading styles for non-heading content.

### Color

- Use semantic color tokens: `text-primary`, `bg-surface`, `border-default` — not raw hex values. (Whether tokens are utility classes, CSS variables, or theme objects, the rule is: name the role, not the value.)
- Ensure sufficient contrast (4.5:1 for normal text, 3:1 for large text)
- Don't rely solely on color to convey information (use icons, text, or patterns too)

## Accessibility (WCAG 2.1 AA)

Accessibility lives in semantic HTML and ARIA, which are platform-agnostic — the markup below is the same whether emitted by JSX, a Vue/Svelte/Angular template, or hand-written HTML. Every component must meet these standards.

### Keyboard Navigation

```tsx
// Every interactive element must be keyboard accessible.
// Prefer native interactive elements — they are focusable and
// keyboard-operable for free.
<button onClick={handleClick}>Click me</button>        // ✓ Focusable by default
<div onClick={handleClick}>Click me</div>               // ✗ Not focusable
<div role="button" tabIndex={0} onClick={handleClick}    // ✓ But prefer <button>
     onKeyDown={e => {
       if (e.key === 'Enter') handleClick();
       if (e.key === ' ') e.preventDefault();
     }}
     onKeyUp={e => {
       if (e.key === ' ') handleClick();
     }}>
  Click me
</div>
```

### ARIA Labels

```tsx
// Label interactive elements that lack visible text
<button aria-label="Close dialog"><XIcon /></button>

// Associate labels with form inputs
<label htmlFor="email">Email</label>
<input id="email" type="email" />

// Or use aria-label when no visible label exists
<input aria-label="Search tasks" type="search" />
```

### Focus Management

Move focus deliberately when content appears or changes (opening a dialog, revealing a panel), and trap focus inside modal surfaces while they're open. The mechanism for running this side effect differs per framework (effects, lifecycle hooks, actions/directives), but the requirement is identical.

```tsx
// React illustration — the principle: focus the dialog's first control on open,
// and keep focus trapped inside it.
function Dialog({ isOpen, onClose }: DialogProps) {
  const closeRef = useRef<HTMLButtonElement>(null);

  useEffect(() => {
    if (isOpen) closeRef.current?.focus();
  }, [isOpen]);

  return (
    <dialog open={isOpen}>
      <button ref={closeRef} onClick={onClose}>Close</button>
      {/* dialog content */}
    </dialog>
  );
}
```

### Meaningful Empty and Error States

```tsx
// Don't show blank screens — give the user context and a next action.
function TaskList({ tasks }: { tasks: Task[] }) {
  if (tasks.length === 0) {
    return (
      <div role="status" className="text-center py-12">
        <TasksEmptyIcon className="mx-auto h-12 w-12 text-muted" />
        <h3 className="mt-2 text-sm font-medium">No tasks</h3>
        <p className="mt-1 text-sm text-muted">Get started by creating a new task.</p>
        <Button className="mt-4" onClick={onCreateTask}>Create Task</Button>
      </div>
    );
  }

  return <ul role="list">...</ul>;
}
```

## Responsive Design

Design for mobile first, then expand. Whatever your styling layer, start from the smallest viewport and add breakpoints upward — utility classes, media queries, and container queries all express the same intent.

```tsx
// Mobile-first responsive (utility-class illustration).
// Plain CSS equivalent: define base styles, then min-width media queries.
<div className="
  grid grid-cols-1      /* Mobile: single column */
  sm:grid-cols-2        /* Small: 2 columns */
  lg:grid-cols-3        /* Large: 3 columns */
  gap-4
">
```

Test at these breakpoints: 320px, 768px, 1024px, 1440px.

## Loading and Transitions

Two principles, framework-independent:

1. **Use skeletons for content loading, not spinners** — they preserve layout and reduce perceived wait.
2. **Use optimistic updates for actions** — reflect the change immediately, then reconcile with the server and roll back on error.

```tsx
// Skeleton loading — translate the markup/animation to your stack.
function TaskListSkeleton() {
  return (
    <div className="space-y-3" aria-busy="true" aria-label="Loading tasks">
      {Array.from({ length: 3 }).map((_, i) => (
        <div key={i} className="h-12 bg-muted animate-pulse rounded" />
      ))}
    </div>
  );
}
```

```tsx
// Optimistic update — TanStack Query illustration. The principle (apply locally,
// snapshot previous state, roll back on error) maps to SWR, Apollo, Pinia,
// RTK Query, or a hand-written mutation just as well.
function useToggleTask() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: toggleTask,
    onMutate: async (taskId) => {
      await queryClient.cancelQueries({ queryKey: ['tasks'] });
      const previous = queryClient.getQueryData(['tasks']);

      queryClient.setQueryData(['tasks'], (old: Task[]) =>
        old.map(t => t.id === taskId ? { ...t, done: !t.done } : t)
      );

      return { previous };
    },
    onError: (_err, _taskId, context) => {
      queryClient.setQueryData(['tasks'], context?.previous);
    },
  });
}
```

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Accessibility is a nice-to-have" | It's a legal requirement in many jurisdictions and an engineering quality standard. |
| "We'll make it responsive later" | Retrofitting responsive design is 3x harder than building it from the start. |
| "The design isn't final, so I'll skip styling" | Use the design system defaults. Unstyled UI creates a broken first impression for reviewers. |
| "This is just a prototype" | Prototypes become production code. Build the foundation right. |
| "The AI aesthetic is fine for now" | It signals low quality. Use the project's actual design system from the start. |

## Red Flags

- Components that try to do too much (split them by responsibility, not by an arbitrary line count)
- Inline styles or arbitrary values that bypass the spacing/color scale
- Missing error states, loading states, or empty states
- No keyboard navigation testing
- Color as the sole indicator of state (red/green without text or icons)
- Generic "AI look" (purple gradients, oversized cards, stock layouts)
- Reaching for a global store when local, lifted, URL, or server state would do

## Verification

After building UI:

- [ ] Component renders without console errors
- [ ] All interactive elements are keyboard accessible (Tab through the page)
- [ ] Screen reader can convey the page's content and structure
- [ ] Responsive: works at 320px, 768px, 1024px, 1440px
- [ ] Loading, error, and empty states all handled
- [ ] Follows the project's design system (spacing, colors, typography)
- [ ] No accessibility warnings in dev tools or axe-core
